# Система учета личных финансов

## 1. Выбранная предметная область
Система представляет собой консольное приложение для контроля и учета личных финансов. Она позволяет отслеживать состояние счетов, группировать движения денежных средств по категориям, фиксировать транзакции и следить за прогрессом достижения финансовых целей.

### Сущности системы (5 сущностей):
1. **User (Пользователь)** - владелец счетов и финансовых целей. Поля: `id`, `name`, `email`.
2. **Account (Счет)** - кошелек или банковский счет пользователя. Связан с пользователем (`userId`). Поля: `id`, `userId`, `name`, `balance`.
3. **Category (Категория)** - классификатор операций (например, "Продукты", "Зарплата"). Поля: `id`, `name`, `type` (Доход/Расход).
4. **Transaction (Транзакция)** - конкретная финансовая операция. Связана со счетом (`accountId`) и категорией (`categoryId`). Поля: `id`, `accountId`, `categoryId`, `amount`, `date`, `description`.
5. **Goal (Финансовая цель)** - накопления на определенную задачу. Связана с пользователем (`userId`). Поля: `id`, `userId`, `title`, `targetAmount`, `currentAmount`.

## 2. Структура папок
Проект организован строго в соответствии с принципами модульной многослойной архитектуры:

* **`bin/`** - содержит файл `main.dart` (точка входа: инициализация БД и запуск Menu).
* **`lib/`** - содержит весь исходный код:
    * `mmm.dart` - публичный экспорт всех необходимых классов и функций.
    * **`src/domain/`** - бизнес-логика: директории `models/` (сущности), `validators/` (чистые функции) и `interfaces/` (абстракции репозиториев).
    * **`src/data/`** - инфраструктура: файл `database.dart` (настройка SQLite) и директория `repositories/` (реализация CRUD).
    * **`src/cli/`** - интерфейс: файлы `menu.dart` (главное меню) и `input_helper.dart` (ввод с валидацией).
* **`test/`** - содержит независимые тесты для каждого слоя (`domain_test.dart`, `data_test.dart`, `validation_test.dart`).

## 3. Разделение на слои и обоснование
Архитектура разделена на три изолированных слоя для обеспечения слабой связанности компонентов, тестируемости и простоты поддержки:

* **Слой Domain (Бизнес-логика)**
    * **Что содержит:** Классы моделей данных (`models/`), чистые функции бизнес-валидации (`validators/`), абстрактные контракты (`interfaces/`).
    * **Обоснование:** Слой полностью изолирован. Он ничего не знает о существовании базы данных (инфраструктуры) и интерфейса пользователя (консоли/stdin). Это позволяет изменять логику хранения или отображения данных, не затрагивая ядро приложения. Тестируется независимо от внешних факторов.
* **Слой Data (Инфраструктура хранения)**
    * **Что содержит:** Логику подключения к SQLite (`database.dart`) и репозитории (`repositories/`), реализующие конкретные SQL-запросы для выполнения CRUD-операций.
    * **Обоснование:** Импортирует только слой `domain`. Он изолирует детали реализации базы данных. Если в будущем SQLite потребуется заменить на JSON-файлы, удаленный сервер или NoSQL, изменения затронут исключительно этот слой, не нарушая работоспособность бизнес-логики и UI.
* **Слой CLI / Presentation (Пользовательский интерфейс)**
    * **Что содержит:** Консольное меню, обработку команд пользователя, вывод таблиц на экран и интерактивный ввод данных.
    * **Обоснование:** Зависит как от слоя `domain` (для использования моделей и валидаторов), так и от слоя `data` (для вызова CRUD-методов репозиториев). Логика UI отделена от бизнес-правил, что позволяет при необходимости беспрепятственно заменить консольный ввод/вывод на графический веб-интерфейс или мобильное приложение Flutter, полностью переиспользовав слои `domain` и `data`.

## 4. Реализованные валидации
Валидация разделена на декларативные чистые функции в слое `domain` и их циклическое применение к потоку ввода `stdin` во вспомогательном модуле `cli/input_helper.dart`.

1. **Обязательное текстовое поле (`validateNotEmpty`)**:
    * Проверяет, что строка ввода не является пустой и не состоит исключительно из пробельных символов.
    * Алгоритм: проверка `value.trim().isEmpty`. В случае ошибки возвращает строку с описанием проблемы, при успехе - `null`.
2. **Форматное/числовое поле операции (`validatePositiveAmount`)**:
    * Гарантирует корректность ввода финансовых сумм и лимитов.
    * Алгоритм: проверяет успешность парсинга строки в `double` через `double.tryParse()`, после чего контролирует, что полученное число строго больше нуля (`value > 0`).

## 5. Перечень тестов
Тестирование слоев реализовано независимо друг от друга в директории `test/`:

1. **`test/domain_test.dart` (Тесты сущностей)**:
    * Проверка сериализации и десериализации данных.
    * Контроль корректности работы методов `toMap()` и фабрик `fromMap()` для каждой из 5 сущностей для предотвращения искажения структуры данных при сохранении в БД.
2. **`test/data_test.dart` (Тесты базы данных)**:
    * Интеграционное тестирование CRUD-операций репозиториев.
    * Запуск тестов во временной SQLite базе данных в памяти (`sqlite3.openInMemory()`).
    * Проверка сценариев: успешное создание записи, чтение списка элементов, обновление полей существующей записи по ID, каскадное удаление связанных сущностей (`ON DELETE CASCADE`).
3. **`test/validation_test.dart` (Тесты бизнес-правил)**:
    * Модульное тестирование чистых функций валидации.
    * Проверка пограничных сценариев (валидный ввод, пустая строка, пробелы, отрицательные числа, нечисловые символы в денежных полях) и ожидаемых ответов об ошибках.
